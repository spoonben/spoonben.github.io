Git

At work we use Git for our version control. It is pretty neat. Git is a distributed revision control and source code management system. It is fast, light, and gets the job done well.It may just be because we are also using GitHub Enterprise along with it that makes it work so well, but it is pretty awesome anyway.

I was used to using Visual Studio with Microsoft Source Safe. The most we did really was checking files in and out of the server, we never did feature branches or true versions. It didn't seem like, at least while we were using it, there wasn't much control over commit history or making quick rollbacks without doing the entire codebase. This was at least my experience. I'm sure there are those out there that use it as their version control and it works like a dream - but for me, not so much.

Now that I am using Git, and using it the correct way, I don't think I could ever go back. I mean, who could stray from the Octocats? Nobody - that's who.

When you know what you are doing it is a safe and effective tool. When you don't know what you are doing, and things aren't setup correctly, it will screw your day up. I could go on here to tell you about how I lost tons of work when I first started using Git, but I won't, it is a sad story. That being said, I am in an environment that was setup by people who knew what they were doing - and I had some great people to teach me how to use it. So now it is pretty nice to use. It is especially nice when you have the correct standards setup. We use most of the regular Git standards for our commit process (here are some from OpenStack) and some of our own stuff.

My Git Config

Here is some of my git config from work. It has some useful aliases and shortcuts:


            [alias]
            co = checkout
            ci = commit
            st = status
            br = branch
            l = log --graph --pretty=format:'%Cred%h%Creset -%C(yellow)%d%Creset %s %Cgreen(%cr) %C(bold blue)<%an>%Creset' --abbrev- commit --date=relative
            po = push origin
            prp = "!f() { git whatchanged origin/master..HEAD; }; f"
            [merge]
            tool = opendiff

So let me explain some of those (that aren't obvious):

  • l (lowercase L) - this will display a pretty sweet interpretation of the git log.
  • prp - basically will show you what is going to go into your pull request.

I like using opendiff as my mergetool, but you can always use your favorite editor. opendiff just seems like the best lightweight option.
